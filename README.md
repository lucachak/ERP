## 📋 Overview

**Cambiomatic** is a web ERP (Enterprise Resource Planning) application focused on solving the daily challenges of a mechanic workshop specialized in **vehicle air conditioning**. Without relying on AI or over-the-top solutions — the system delivers what really matters: **speed, stability, and practical control** over service orders, customers, inventory, and financial flow.

---

## ✨ Features

### 📄 Service Orders (SO)
- Creation of SO with simultaneous registration of customer and vehicle (new or existing).
- Unique identifier automatically generated by UUID.
- Addition of **services** and **parts** to an open SO.
- Automatic calculation of totals (services + parts + shipping).
- Integrated technical checklist (gas pressure, cabin filter, compressor, sanitization).
- SO closing with delivery date and automatic **90-day warranty**.
- Formatted SO printing for the customer.
- Paginated search by number, customer, license plate, etc.

### 🗂️ SO Kanban
- Visualization of SOs in a Kanban board with 4 status columns:
  - **Open** → **Waiting for Part** → **In Service** → **Closed**
- Status update with drag-and-drop (via HTMX, no reload).
- Filter by term directly on the board.

### 👥 Customers & Vehicles
- Complete customer registration (name, tax ID, ID, address, contacts).
- Vehicle history per customer (model, license plate/chassis, year, prisma).
- Search and filter with paginated listing.

### 🔧 Standard Services
- Table of pre-registered services with default values.
- Reusable in SO item creation (with the possibility to override the value).

### 📦 Inventory (Parts)
- Parts registration with code, description, quantity, and selling price.
- Automatic inventory reduction when adding a part to an SO.
- Automatic inventory reversal when removing a part from an SO.
- **Critical stock alert** on the Dashboard (quantity ≤ 2 units).

### 💰 Cash Flow (Finance)
- Recording of financial inflows and outflows.
- Filter by month/year.
- Balance calculation: `Inflows − Outflows`.
- Extra fields: status (PENDING/PAID), due date, payment date, category, payment method.
- **One-click settlement** for pending accounts.
- **Export to CSV** (Excel compatible, with UTF-8 BOM for accents).

### 📊 Dashboard (Admin Panel)
- Current month KPIs: revenue, open SOs, stock alerts, average service time.
- **Donut Chart** — distribution of SOs by status.
- **Bar Chart** — revenue from the last 6 months.
- Table of the 5 most recent SOs issued.

### 📅 Schedule
- Registration and listing of appointments.

### 📈 Commission Report
- Grouping of services performed by technician in the selected month.
- Calculation of total hours and total value per technician.

### 👤 User Management
- User creation with two profiles: **ADMIN** and **Mechanic**.
- User profile with password change.
- Quick user creation via modal (partial response via HTMX).

### 🔐 Security
| Feature | ADMIN | Mechanic |
|---|---|---|
| Dashboard, Cash Flow, Reports | ✅ | ❌ |
| Users, Backup, Schedule, Kanban | ✅ | ❌ |
| Service Orders (listing/details) | ✅ | ✅ |
| Customers, Inventory, Standard Services | ✅ | ✅ |

### 💾 Backup
- Database backup generation via `pg_dump` from the web interface itself.
- Download the `.sql` file directly through the browser.

## 🚀 How to Run Locally

### Prerequisites

- **Java 21+** — [Download](https://adoptium.net/)
- **Maven 3.9+** — or use the included `./mvnw` wrapper
- **PostgreSQL** — local instance or free account on [Neon](https://neon.tech/)

### 1. Clone the repository

```bash
git clone https://github.com/lucachak/Gen.git
cd Gen
```

### 2. Configure environment variables

Create a `.env` file in the root (or export variables in your terminal):

```bash
export DATABASE_URL=jdbc:postgresql://<host>/<database>?sslmode=require
export DATABASE_USER=<your_user>
export DATABASE_PASS=<your_password>
```

> **Note:** `application.properties` already supports these variables with default fallback values. For production, **always use environment variables**.

### 3. Run the application

```bash
# With Maven Wrapper (recommended)
./mvnw spring-boot:run

# Or directly with Maven
mvn spring-boot:run
```

The application will be available at **http://localhost:8080**

### 4. Default login (automatically created on 1st run)

| Field | Value |
|---|---|
| User | `dalva` |
| Password | `dalva` |
| Profile | `ADMIN` |

> ⚠️ **Change the default password immediately** after first access at **/perfil**.

## 🐳 Deploy with Docker

### Local image build

```bash
docker build -t cambiomatic .
```

### Run the container

```bash
docker run -p 8080:8080 \
  -e DATABASE_URL=jdbc:postgresql://<host>/<db>?sslmode=require \
  -e DATABASE_USER=<user> \
  -e DATABASE_PASS=<password> \
  cambiomatic
```

The `Dockerfile` uses **multi-stage build**:
- **Stage 1 (`build`):** Maven + JDK 21 Alpine — compiles and packages the `.jar`
- **Stage 2 (`run`):** JRE 21 Alpine — minimal image for production

## ☁️ Production Deploy (Render)

1. Create a new **Web Service** on [Render](https://render.com/) pointing to this repository.
2. Define the environment variables (`DATABASE_URL`, `DATABASE_USER`, `DATABASE_PASS`) in the Render dashboard.
3. Render will detect the `Dockerfile` and build the image automatically.

### Keep-Alive (free plan)

Render's free plan hibernates the application after inactivity. The `keep_alive.py` script performs periodic pings (every 60–90 seconds) to keep the application active:

```bash
python3 keep_alive.py
```

> To automate, configure a scheduled **GitHub Actions** (`cron`) that runs this script.

## 🏗️ Architecture Decisions

- **MVC with Thymeleaf:** Server-side rendering for maximum simplicity and SEO. No unnecessary SPA complexity for an internal system.
- **HTMX:** Partial page updates (add item, change status in kanban, toasts) without writing custom JavaScript — keeping the backend as the single source of truth.
- **Flyway:** Database schema versioning. Ensures any new deploy starts from a consistent schema.
- **Lombok:** Eliminates boilerplate getters/setters/constructors in models and services.
- **Records (DTOs):** Use of Java Records for immutable and compact DTOs.
- **CSRF disabled:** Required for HTMX to work correctly with partial forms. In production, consider configuring a CSRF strategy compatible with HTMX.
